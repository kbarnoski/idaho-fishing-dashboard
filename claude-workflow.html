The trap you just hit

Your iPhone work sat on claude/connect-resonance-project-nIxHM while the laptop and the agents kept pushing to main. Nothing was lost — but the two sides had drifted and needed a deliberate merge. That is the normal state of multi-device work. The habits below keep the drift small and boring instead of large and scary.

The five basics

  1. Pull before you start. Pull before you push. The other two writers may have moved while you were away. A 2-second fetch prevents a 20-minute conflict.
  2. Push small and often. Your phone and laptop can only see what's on GitHub. Unpushed work is invisible to your other device — and to you, an hour later.
  3. Know which branch your work is on. Phone/web work is always on a claude/… branch. It is not on main until you merge it. Check git branch -r | grep claude when something "isn't showing up."
  4. One change → one push → one build. Each push to main triggers exactly one Vercel deploy. Batch a coherent change, push once.
  5. Don't let three writers collide silently. Agents push to main every 2 hours. If you're also editing main, rebase often so you're never more than a couple commits out of sync.

Daily playbooks · copy & paste

① Start a laptop sessionSee what the phone & agents did while you were gone
# what changed on main since I last looked?
git fetch origin
git log --oneline HEAD..origin/main
# any phone/web work waiting on a branch?
git branch -r | grep claude
# catch up (replays your local commits on top, if any)
git pull --rebase origin main
② Pull in your phone's workThe exact move we just did — merge a claude/… branch into main
git fetch origin
git branch -r | grep claude            # find the branch name
git log --oneline main..origin/claude/<slug>   # preview the commits
git merge --ff-only origin/claude/<slug>       # clean merge, no merge commit
git push origin main
# If --ff-only fails (you ALSO changed main), the branch diverged:
#   git merge origin/claude/<slug>   → makes a merge commit, resolve conflicts, push
③ Before every pushOne habit that prevents almost all conflicts
git pull --rebase origin main && git push origin main
④ "It says diverged — ahead X, behind Y"Both sides moved. Replay your commits on top of theirs.
git fetch origin
git rev-list --left-right --count origin/main...HEAD   # behind <tab> ahead
git pull --rebase origin main
# fix conflicts if git stops, `git add` them, `git rebase --continue`
git push origin main

From here to hundreds of projects & agents

01One repo = one source of truth. Every project is a GitHub repo. Laptop, phone, and agents all reconcile there. Never edit a file in two places without GitHub in the middle.
02Decide who owns main per repo. If an agent regenerates a file daily (like the fishing dashboard's index.html), treat the agent as the owner — your hand-edits there will be overwritten. Put human work elsewhere or in the agent's prompt.
03Naming conventions scale, memory doesn't. Branch claude/<project>-<task>, name routines <repo> — <job>. When you have 100 of each, the name is the only thing that tells you what it does.
04Keep a registry. A single table (repo → URL → owning agents → schedule → budget) is what turns "a pile of projects" into "a fleet you can review." This dashboard is v1 of that registry.
05Respect the fire budget. 15 routine fires / 24h account-wide. At scale, schedule deliberately — stagger crons, prefer daily over hourly, reserve headroom for ad-hoc runs.
06Review from one pane. claude.ai/code/routines for agents; GitHub's commit feed per repo for output. As the count grows, a daily "what did my fleet ship" rollup (itself an agent) becomes the thing you actually read.
